# 📋 代码审查检查清单

## PR提交前自检

### 🔧 代码质量检查
- [ ] 代码遵循SwiftLint规范
- [ ] 没有编译警告或错误
- [ ] 所有TODO标记都有明确的后续计划
- [ ] 删除了调试代码和注释掉的代码
- [ ] 变量命名清晰且符合命名规范

### 🧪 测试相关
- [ ] 新增功能有相应的单元测试
- [ ] 修改现有功能后测试仍然通过
- [ ] 测试覆盖率没有明显下降
- [ ] 测试用例命名清晰描述测试目的
- [ ] Mock对象使用合理

### 🏗️ 架构合规性
- [ ] 遵循MVVM+Coordinator架构模式
- [ ] 没有违反依赖注入原则
- [ ] ViewModel不直接操作UI
- [ ] 数据流向清晰单一
- [ ] 组件职责单一，符合SOLID原则

### 📱 UI/UX一致性
- [ ] 遵循Design System规范
- [ ] 使用统一的颜色、字体、间距Tokens
- [ ] 响应式布局适配不同屏幕尺寸
- [ ] 交互反馈符合用户体验标准
- [ ] 无障碍访问性考虑充分

## Reviewer审查要点

### 🔍 功能正确性
- [ ] 功能实现符合需求规格
- [ ] 边界条件处理完整
- [ ] 错误处理机制健全
- [ ] 异常情况下有适当降级方案
- [ ] 性能影响在可接受范围内

### 🔒 安全性检查
- [ ] 没有硬编码敏感信息
- [ ] 数据传输使用安全协议
- [ ] 用户输入进行了适当验证
- [ ] 权限申请理由充分明确
- [ ] 符合App Store审核指南

### 🚀 性能优化
- [ ] 没有明显的内存泄漏风险
- [ ] 图片资源压缩和格式优化
- [ ] 网络请求有适当的缓存策略
- [ ] 列表渲染有合理的复用机制
- [ ] 启动时间和页面加载时间合理

### 📚 文档完整性
- [ ] 公共接口有清晰的注释
- [ ] 复杂业务逻辑有必要的说明
- [ ] 新增组件有使用示例
- [ ] Breaking Changes有迁移说明
- [ ] README相应部分已更新

## 质量门禁标准

### ⚠️ 阻止合并的情况
- CI构建失败
- 测试覆盖率低于80%
- SwiftLint严重违规
- 存在已知的安全漏洞
- 性能回归超过10%

### 🔄 需要讨论的情况
- 架构设计有重大调整
- 影响现有核心功能
- 技术债增加较多
- 第三方依赖变更
- 用户体验有显著变化

## 审查流程

1. **开发者自检** - 提交PR前完成检查清单
2. **自动化检查** - CI流水线自动验证
3. **同行审查** - 至少一名团队成员审查
4. **最终批准** - Tech Lead或项目经理批准
